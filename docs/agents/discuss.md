### 一、 方案的可行性与核心原理

对于不支持原生接收 `application/pdf` 格式的大模型，将 PDF 每页渲染为图片传入，原理如下：

1. **多模态输入转换**：
   * 将 PDF 的第 $1, 2, \dots, n$ 页，在本地利用 Node.js 库渲染成 high-resolution 的图片（如 `image/png` 或 `image/jpeg`）。
   * 将这些图片转换成 Base64 编码，以 OpenAI 标准的 `image_url` 格式组装进消息体中（可按页顺序交织传入）。
2. **大模型的视觉理解**：
   * 视觉大模型（如 MiMo-V2.5）在接收到这些页面图片后，利用其 Vision 编码器，能像人类一样直接“看”到页面上的所有元素。
   * 这不仅包含了文字内容，更包含了您最关心的**红框标注、表格网格、文字与图片的相对位置**等空间排版信息。

---

### 二、 方案的优缺点分析

#### 🟢 优点

1. **视觉信息零丢失**：
   * PDF 中的表格排版不会因“降维纯文本”而错乱。
   * 报告中“文字后面紧跟的现场隐患照片”可以被大模型一眼看清并准确关联，极大降低了“错位切片”的概率。
   * 现场安全员在 PDF 上画的**红框、黄线、圈注**，大模型可以通过视觉注意力机制（Attention）精准捕捉。
2. **极高的模型兼容性**：
   * 该方案完全不需要大模型本身具备 PDF 的解析和解包能力。
   * 只要模型兼容 OpenAI 的 `image_url`（如 MiMo-V2.5, GPT-4o, Qwen-VL, DeepSeek-VL 等），就可以无缝通用，摆脱了对具体模型供应商的强绑定。

#### 🔴 缺点与挑战

1. **Token 消耗（费用）增加**：
   * 图像的 Token 消耗远大于纯文本。以 OpenAI 的计费为例，一张高分辨率的图片大约会消耗 765 ~ 1105 个 Token，如果是 10 页的 PDF，光图片就会占用约 1 万 Token。
2. **运行性能与传输耗时**：
   * 本地 Node.js 渲染 PDF 页面为图片需要消耗 CPU 算力和时间（通常每页渲染需 200ms ~ 500ms）。
   * 传输大体积的 Base64 图片数据包需要更宽的网络带宽，接口响应时间（TTFT）会变长。
3. **多页文档上下文限制**：
   * 如果 PDF 报告长达数十页甚至上百页，一次性把所有页面图片塞入 prompt 会导致模型上下文（Context Limit）或单次请求图片数量上限超载。不过对于通常只有几页的“安全检查报告”，这完全不是问题。

### 三、 在 Node.js 中的实现路径

如果我们要落地该方案，在 Node.js 中非常容易实现。

#### 1. 选用纯 JS 渲染库（无需安装复杂的系统二进制依赖）

在 Node.js 中，为了避免安装 `graphicsmagick` 或 `ghostscript` 等系统级依赖（这些在最终生产环境部署时往往会带来极大的运维麻烦），推荐使用纯 JavaScript 实现的库：

* **`pdf-img-convert`** 或 **`pdf2pic`（配置纯 JS 渲染器）**：
  它们利用了 Mozilla 的 `pdfjs-dist`（即浏览器里渲染 PDF 的引擎）在 Node.js 环境中配合虚拟 Canvas，将 PDF 的每一页直接渲染输出为 PNG 格式的 `Buffer`。

#### 2. 代码逻辑示意：

```javascript
import pdfHtmlConverter from 'pdf-img-convert';

// 将 PDF 文件直接转化为 Base64 图片数组
export async function convertPdfToImages(pdfPath) {
  // 渲染配置，可以指定 DPI（分辨率）以保证图片中的小字清晰
  const outputImages = await pdfHtmlConverter.convert(pdfPath, {
    width: 1024, // 限制宽度，平衡清晰度与体积
  });
  
  // outputImages 是一个 Uint8Array 数组，每个元素代表一页的图片数据
  return outputImages.map((imgBuffer) => {
    return {
      type: 'image',
      data: Buffer.from(imgBuffer).toString('base64'),
      mimeType: 'image/png'
    };
  });
}
```

### 一、 为什么这种双路并行方案效果明显？

在实际处理安全报告（尤其是 PDF）时，大模型纯视觉识别面临几个痛点，而“无损提取文本”刚好能完美互补。

#### 1. 规避视觉 OCR 字符识别错误（“字符级校对”）

* **视觉的痛点**：多模态模型在读取图片时，对于**英文编号、专业规范代码、日期数字**容易出现“看错”的幻觉。例如：容易把配电箱编号 `SNEI-Ⅱ-016` 误读为 `SNEI-11-016`，或者把 `SH/T3097-2017` 中的 `T` 认成 `I`。
* **双路的优势**：对于非扫描版的电子 PDF，底层纯文本提取库（如轻量级的 `pdfjs-dist` 文字层提取）拿到的字符是 **100% 字符级无损且精准** 的。把这个文本喂给大模型作为“参考字库”，大模型在生成 JSON 时，可以直接复制这段文字，**消除了任何字符拼写幻觉**。

#### 2. 解决印章、下划线、红框遮盖文字的问题（“去噪能力”）

* **视觉的痛点**：安全报告上经常有**单位公章、红色警告印章、手写签名、或者安全员现场画的框线**。这些视觉元素如果正好覆盖在正文上，多模态模型的视觉通道会受到极大的噪声干扰，导致文字识别中断或变乱码。
* **双路的优势**：底层的电子文本流完全不受视觉干扰（盖了章，字依旧在底层的 XML/文本流里）。文本通道提供“字”，视觉通道提供“这里有个红框/公章”的语义，模型融合后可以得出最精准的结论。

#### 3. 极大地提高“隐患边界切片”的逻辑判断力

* **双路的优势**：纯文本通道由于保留了原文档的换行符、制表符等逻辑边界，可以作为大模型进行段落切分的“骨架”；而视觉通道则负责提供“血肉”（如确认某张隐患图片属于哪一段文字）。**骨架+血肉**结合，能够让大模型在一对多提取时，对隐患问题边界的切分达到 99% 的准确率。

---

### 二、 方案的局限性与前置条件

虽然效果极其明显，但它有一个核心的前提：**PDF 必须是电子版（Native PDF），而不是纸张拍照或扫描版 PDF。**

1. **电子版 PDF**：
   * 文字可以直接被代码读取，此方案效果爆表。
2. **扫描版 PDF（图片转的 PDF）**：
   * 底层没有文本流，本地文本提取器只能提取出空白。
   * 这时“文本通道”失效，依然必须全靠多模态模型的 Vision OCR。但只要保留这个架构，当用户上传电子版时，就能立刻享受双路优势。
