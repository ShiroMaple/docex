# DocEx V2.5 生产级可观测性与日志运维指南

本项目已全面引入以 `pino` 和 `pino-roll` 为核心的结构化日志与全链路 Trace 追踪系统，用以取代传统的 `console.log`，支撑高并发及集群部署下的行为审计、异常告警与调用排查。

---

## 📂 一、 日志系统架构设计

1. **日志引擎**：采用原生 ESM 环境下最高效的 `pino` 库，保障极低耗时的流式写入。
2. **日志轮转 (`pino-roll`)**：在生产环境 (`NODE_ENV === 'production'`) 下配置了按天轮转的磁盘日志管道：
   - 存储路径：`/var/www/docex/logs/` (或本地项目根目录 `logs/`)
   - 轮转频率：**每天 (daily)** 自动切割新文件。
   - 文件限制：单文件上限为 **`10MB`**，超出自动切割（自增序列后缀如 `docex.1.log`）。
   - 自动建包：若 logs 目录不存在，启动时自动创建。
   - 版本控制：`logs/` 文件夹已加入 `.gitignore`，防止开发日志污染远端代码库。
3. **Trace 链路追踪**：
   - 基于 Node.js `AsyncLocalStorage (ALS)` 机制，当 API 请求进入时，最外层路由高阶包装器 `withLogging` 自动为请求分配唯一的全局 `traceId`。
   - 在日志输出时，Pino 通过 `mixin()` 自动、隐式地将此 `traceId` 合并进输出 JSON 中，使底层服务（如 WPS、飞书、LLM 客户端）产生的日志可以按单次 HTTP 请求实现完整追踪。

## 📐 二、 结构化日志事件词典 (Event Dictionary)

日志输出为单行 JSON，机器索引友好。其核心字段如下：
- `level`: 日志级别（`30` = INFO, `40` = WARN, `50` = ERROR, `60` = FATAL）
- `traceId`: 链路 Trace ID
- `event`: 大写蛇形字符串，用于机器过滤与报警

### 1. Ingress / Egress 路由与服务追踪
| 事件键名 (`event`) | 对应操作 / 含义 |
| :--- | :--- |
| **`API_INGRESS`** | API 接口接收到请求，包含 Method 与 Path |
| **`API_SUCCESS`** | 接口请求正常返回，包含状态码与耗时 `durationMs` |
| **`API_EXCEPTION`** | 接口未捕获的严重异常，输出完整堆栈，并给前端响应 500 JSON 错误 |

### 2. 隐患解析 (大模型客户端)
| 事件键名 (`event`) | 对应操作 / 含义 |
| :--- | :--- |
| **`LLM_EXTRACTION_START`** | API 路由开始调用 LLM 解析服务，输出待提取的文档 MD5 与字段个数 |
| **`LLM_STRUCTURED_OUTPUTS_START`** | LLM 客户端开始请求 Structured Outputs (`json_schema` 严格模式) |
| **`LLM_JSON_MODE_FALLBACK`** | (警告) 模型不支持严格 Schema，自动降级为 `json_object` 兼容提取模式 |
| **`LLM_EXTRACTION_SUCCESS`** | 文档解析提取成功，包含已消耗的输入、输出、总 Token 数与识别行数 |
| **`LLM_JSON_PARSE_ERROR`** | (错误) 模型未返回合法 JSON，日志自动包含错误原因与残缺的原始吐出文本 `rawContent` |
| **`LLM_API_ERROR`** | (错误) 网络超时或接口秘钥报错等模型 API 底层崩溃 |

### 3. 数据推送 (多维表格对齐网关)
| 事件键名 (`event`) | 对应操作 / 含义 |
| :--- | :--- |
| **`FILE_RECORD_DELETED`** | 用户在前端队列手动清理并删除特定 MD5 文档与物理预处理文件 |
| **`WPS_APPEND_RECORDS_START`** / **`FEISHU_APPEND_RECORDS_START`** | 开始向 WPS / 飞书数据表写入核对完成的数据 |
| **`WPS_APPEND_RECORDS_SUCCESS`** / **`FEISHU_APPEND_RECORDS_SUCCESS`** | 写入多维表格成功，返回记录数与行插入确认 |
| **`WPS_CREATE_FIELD_START`** / **`FEISHU_CREATE_FIELD_START`** | 动态在云端表格中创建缺失的数据列字段 |

## 🛠️ 三、 运维与排查命令指南

生产环境的日志是压缩的单行 JSON，直接阅读较为费力。我们建议在命令行管道中配合 **`npx pino-pretty`** 将 JSON 实时格式化为具备色彩、时间线及关键高亮的友好终端文本。

### 1. 实时跟踪服务器最新日志 (Tail 模式)
在终端中执行以下命令，实时查看系统动态：
```bash
tail -f /var/www/docex/logs/docex.1.log | npx pino-pretty
```

### 2. 追踪特定请求的完整生命周期 (Trace 过滤)
当某个操作报错，前端弹窗或 Response 头部返回了 `traceId` (如 `cda1d4ba-2ac6-4759-84c7-5765af806f32`)，可使用 `grep` 进行链路提取，并使用 `pino-pretty` 渲染：
```bash
cat /var/www/docex/logs/docex.1.log | grep "cda1d4ba-2ac6-4759-84c7-5765af806f32" | npx pino-pretty
```

### 3. 监控大模型调用与 Token 损耗
实时观察大模型调用成功或失败状态，以及详细的 Token 消耗：
```bash
tail -f /var/www/docex/logs/docex.1.log | grep -E "LLM_" | npx pino-pretty
```

### 4. 追踪数据对齐网关推送动作
实时观察数据是否被推送至 WPS 表格或飞书：
```bash
tail -f /var/www/docex/logs/docex.1.log | grep -E "APPEND_RECORDS" | npx pino-pretty
```

## 🔒 四、 安全红线与防御保护

1. **密钥自动混淆**：
   在 `src/lib/logger.js` 的 `redact` 配置中，已将所有外部服务传入的 `apiKey`、`appSecret`、`client_secret` 等高危参数声明为物理拦截。即使不慎将其作为整个结构化对象输出，日志中该字段的值也会被统一置换为 `[REDACTED]`。
2. **爆盘防御**：
   提取过程中产生的 PDF/Word 页面 Base64 图像编码被严格排除在日志文件之外。所有多模态图片二进制数据字段在入日志前会被自动替换为 `[REDACTED]`，保护服务器磁盘空间不被瞬间占满。
3. **临终遗言捕捉**：
   系统进程全局注册了 `uncaughtException` 和 `unhandledRejection`。当发生导致 Node 进程崩溃的致命错误时，系统会在强制退出（`process.exit(1)`）前触发 `logger.flush()` 同步刷新缓冲区，确保崩溃现场完整记入日志。
